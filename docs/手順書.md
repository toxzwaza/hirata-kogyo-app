# hirata-kogyo-app 手順書（解析・運用・追加開発ガイド）

平田工業向けのWebアプリケーション（Laravel + Inertia + Vue3）の機能・構造・運用ルールをまとめたドキュメント。
本書は今後の追加開発・修正の起点として、コードを読み解くための地図となることを目的とする。

---

## 1. アプリ概要

平田工業（鋳物仕上げ加工）の以下業務をまとめて電子化する社内Webアプリ。

| 領域 | 内容 |
|------|------|
| 作業実績 | スタッフが図番・作業方法ごとに作業時間と出来高（良品/不良）を記録。不良は内訳（不良種別×数量）でブレークダウン。 |
| 請求管理（スタッフ→自社） | 期間締めで作業実績を集計し、スタッフ個別の請求書を自動生成・PDF発行。 |
| 請求管理（自社→委託元） | 確定済みのスタッフ請求書を束ねて客先請求書を生成。差額調整・消費税込みで一括PDF出力。 |
| マスタ管理 | 客先・図番・作業方法・作業単価（履歴管理）・スタッフ・不良種類の6種のマスタを管理。 |
| スマホ入力 | 作業者が現場からスマホで作業実績を入力するための専用レイアウト。`?staff_id=N` で自動ログイン可能。 |

### 4.3 認証方式
- `config/auth.php`：guard `web` / provider `staff` → **Userテーブルではなく `staff` テーブルで認証**。
- `Staff` モデルは `Authenticatable` を継承し `getAuthIdentifierName()` で `login_id` を返す。
- パスワードはミューテータで自動ハッシュ化。
- `admin_flg` はスタッフに管理者権限を持たせる想定（平田敦士のみ `true`）。

## 6. ドメインロジックの要点

### 6.1 作業単価の履歴管理
- `work_rates` は図番×作業方法×期間で単価を保持。`effective_from` / `effective_to` で有効期間を管理し、**過去分は改変しない**。
- 新しい単価を登録すると、`WorkRateController@store` が既存の有効な単価の `effective_to` を新単価の前日に更新する（期間の自動閉鎖）。
- `Drawing::getEffectiveWorkRate($workMethodId, $date)` で任意時点での有効単価を取得。
- `active_flg=false` の単価に紐づいた `work_records` は `RelinkWorkRecordsService` で「有効な単価」に再紐付け可能（`POST /work-rates/relink-work-records`）。

### 6.2 作業実績の登録ルール
- 登録時に `start_time` 時点で有効な `work_rate` を自動取得し `work_rate_id` に保存（**単価スナップショット**）。
- `work_minutes` は `end_time - start_time` の分数で自動算出。
- 不良数（`quantity_ng`）の合計と不良内訳（`defects[*].defect_quantity`）の合計が **一致しないと例外**。
- `WorkRecord::isOvertime()` は「開始時刻の時が 17 以上なら残業」と判定（個人事業主の残業単価に使用）。
- 請求書明細（`StaffInvoiceDetail`）が既に存在する作業実績は **編集・削除不可**（`isInvoiced()`）。

### 6.3 スタッフ請求書（StaffInvoiceService）
1. `createInvoice(staff_id, period_from, period_to)`：
   - 期間内で未請求（`staffInvoiceDetails` を持たない）かつ `workRate.active_flg=true` の作業実績を取得。
   - 実績ごとに `total_weight = (quantity_good + quantity_ng) × weight_per_unit`。
   - 単価は `getRateForStaff($staff, $isOvertime)`：正社員=`rate_employee`、個人事業主は残業かどうかで `rate_overtime`/`rate_contractor`。
   - `amount = weight × unit_price` を明細に**スナップショット**保存 → 後の単価変更に影響されない。
   - `tax_type=taxable` のスタッフのみ消費税10%（`floor` 切り捨て）。非課税は `tax=null`。
2. `fixInvoice` / `unfixInvoice`：
   - `draft → fixed`。客先請求書に紐付いていると `unfix` 不可。
3. 請求書番号：`STAFF-YYYY-NNN`（年単位で連番）。
4. PDF（`StaffInvoiceController@pdf`）：mPDF で 1 スタッフ 1 ファイル出力。明細のスナップショットを優先使用。

### 6.4 客先請求書（ClientInvoiceService）
1. 確定済み（`fixed`）かつ他の客先請求書に未使用のスタッフ請求書のみ対象。
2. `subtotal` = 選択スタッフ請求書の `total` 合計、`tax` = subtotal × 10%（floor）、`adjustment_amount` は後から編集可能。
3. `total = subtotal + adjustment_amount + tax`。
4. 現行実装では、客先は **`株式会社○○`（`firstOrCreate`）** で固定。→ 将来的に画面から選択できるよう改修予定ポイント。
5. `fixInvoice` で確定。確定済のみ PDF 出力可能。
6. PDF は **1枚目：スタッフごとに集約した金額リスト**、**2枚目以降：スタッフごとの明細内訳** の構造を `client-invoice.blade.php` 1本で `type` 切替して描画。
7. 請求書番号：`CLIENT-YYYY-NNN`。

### 6.5 支払い期限 (`payment_due_date`)
- `Carbon::createFromFormat('Y-m-d', ..., 'UTC')` で保存し、`show` では `format('Y-m-d')` で返す
  → **JSON化時に前日にズレる現象** を防ぐための実装。日付を扱う追加開発時はこの方針に倣うこと。

### 6.6 スマホ入力
- `Layouts/MobileLayout.vue` を使用。
- URL `/staff/work-records/create?staff_id=N` を QR 等で配布する想定。
- 分は 5 分単位に丸め、0〜5 時は入力不可（現場運用：始業前の誤登録防止）。
- 登録成功後は同じ画面に留まり、連続登録できる。

## 9. 運用メモ

### 9.1 請求書系の「スナップショット」原則
`staff_invoice_details.unit_price / amount` や `work_records.work_rate_id` は**作成時点の値を固定保存**している。
単価マスタや図番の重量を後から変えても、過去の請求書の金額は変わらない。
追加開発時も**過去請求書の金額は絶対に再計算しない**こと（監査証跡が崩れる）。

### 9.2 ステータス遷移
- スタッフ請求書：`draft` → `fixed` → `paid`
- 客先請求書：`draft` → `fixed` → `issued`
- 客先請求書の PDF 出力は `fixed`/`issued` のみ可、削除は `draft` のみ可。

### 9.3 インボイス登録番号
現在 DB に登録番号カラムは無いが、スタッフ側の `tax_type` で課税/非課税を持っている。
将来、適格請求書発行事業者番号を追加する場合は `staff` と `clients` の両方に `invoice_registration_number` 等を追加する設計で検討。

### 9.4 客先請求書の客先固定
`ClientInvoiceController@store` で `Client::firstOrCreate(['name' => '株式会社○○'])` となっているため、
**現状は複数客先に請求書を切ることができない**。追加開発時の最優先課題。

### 9.5 スマホ自動ログイン
`/staff/work-records/create?staff_id=N` はパスワードレスでログインする危険な実装。
本番で公開URLになる場合は、
- ステッカーQRの再発行運用
- IP制限・ワンタイムトークン化
のいずれかで早めに締める想定。

### 9.6 セッションタイムゾーン
`work_records.start_time / end_time` は `datetime`（タイムゾーン無し）。
`payment_due_date` のみ UTC 0:00 で保存して表示時に `format('Y-m-d')` する特殊運用。
**日時系カラムを追加する時はここを必ず合わせる**。

## 10. 追加開発の着手順（推奨）

優先度は既知課題と実運用インパクトから推奨。

### Phase 1：基盤固め（小規模だが重要）
1. **客先請求書の客先選択UI化**（`ClientInvoiceController@store` の固定客先を廃止し、`Create.vue` で `client_id` を選択）。
2. **スマホ自動ログインの安全化**（ワンタイムトークン or パスワード必須化）。
3. **/__debug ルートの本番無効化**（`APP_ENV !== 'production'` のガード）。
4. **Users テーブルの扱いを整理**（現状 Staff を Authenticatable にしており users テーブルはBreezeの副産物。マイグレーションからの削除可否を検討）。
