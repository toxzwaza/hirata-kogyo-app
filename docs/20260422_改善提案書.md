## 0. エグゼクティブサマリー

本システムは現場（スマホ）〜請求書発行までの業務を一気通貫で扱う社内Webアプリとして、基本骨格は良好である。一方で、以下3点の早期対応を強く推奨する。

1. **セキュリティ・本番運用リスクの解消**：スマホ自動ログインの平文ID渡し、本番でも応答する `/__debug` ルート、Breezeの残骸。
2. **Controller肥大と重複ロジックのService層への集約**：`WorkRecordController` / `ClientInvoiceController@pdf` の2ファイルで約900行あり、同一処理が3か所（PC登録／スマホ登録／PC更新）に散っている。
3. **Laravel 9・PHP 8.0 のEoL対応**：Laravel 9は2024年2月にサポート終了済。本番（hirata-industry.com）で稼働中のため、バージョンアップ計画の着手が必要。

以降、A〜Eの観点で改善提案をまとめる。**Phase分けと工数の目安は第7章**を参照。

## 2. 改善提案【A：セキュリティ・本番運用リスク】

### A-1. `/__debug` ルートを本番で無効化（最優先）

- **現状**：`routes/web.php:27-35` で `GET /__debug` が middleware なしで公開。本番環境でも叩けば内部情報（scheme/host/header）を返す。
- **提案**：`APP_ENV !== 'production'` でのみ登録、もしくは `local`/`development` ミドルウェアでガード。調査が済んだら削除。

### A-2. スマホ自動ログインの安全化（最優先）

- **現状**：`WorkRecordController.php:106-120` の `createForMobile()` が `?staff_id=N` を受け取り、**パスワードチェックなしで `Auth::login($staff)`** している。コード内コメントにも「危険だが、とりあえず」と明記。
- **リスク**：URLを知っていれば誰でも任意のスタッフに成りすませる。請求対象の作業実績が改ざんされうる。
- **提案（2段構え）**：
  - **短期**：`staffs` テーブルに `mobile_login_token`（ユニーク、一定間隔で再発行）を追加し、`?token=xxxx` で認証する。QRステッカー運用はそのまま継続。
  - **中期**：PWA化とあわせて、スタッフ個別のPINコード入力（4〜6桁）を併用。
  - 追加で、**接続元IP制限**（社内ルータ／現場の固定グローバルIP）をミドルウェアで実装。

### A-3. 権限チェック（認可）の不在

- **現状**：全FormRequestの `authorize()` が `return true;`（例：`StoreWorkRecordRequest.php:16-19`）。`Staff` モデルに `admin_flg` カラムがあるのに活用されていない。
- **提案**：
  - `Policy` を `StaffInvoicePolicy` / `ClientInvoicePolicy` / `StaffPolicy` / マスタ系で整備。
  - 「請求書の確定／解除／削除」「スタッフマスタの編集」「単価の再紐付け」は `admin_flg=true` のみ。
  - 一般スタッフは「自分の作業実績の登録・閲覧」のみ許可。
  - FormRequest の `authorize()` を `return $this->user()->can('update', $this->route('work_record'));` 等に書き換え。

### A-4. Breeze副産物の整理

- **現状**：
  - 認証は `Staff` モデルで行っているのに、`users` テーブルのマイグレーション・Authコントローラ一式・`Pages/Auth/Register.vue` 等が残る。
  - **一般社員でも `/register` でアカウント作成できてしまう**（`routes/auth.php`）。
- **提案**：
  - `routes/auth.php` の register／forgot-password／reset-password 系ルートを無効化（もしくは admin only）。
  - `users`・`password_resets`・`personal_access_tokens` テーブルが本当に不要か確認のうえ、`drop_unused_*` マイグレーションで削除。
  - `Pages/Auth/Register.vue` / `ConfirmPassword.vue` / `VerifyEmail.vue` を削除。

### A-5. 環境変数の本番ガード

- **現状**：`.env.example:4` が `APP_DEBUG=true`。本番 `.env` では `false` になっていること前提だが、明示的な確認手順がない。
- **提案**：`手順書.md` に「デプロイ後に `php artisan about` or `APP_DEBUG` を確認」の項目を追加。あわせて `.env.example` のデフォルトを `APP_DEBUG=false` にして、開発時のみ上書きする運用へ。

## 3. 改善提案【B：リファクタリング】

### B-1. `WorkRecordController` の重複除去（優先）

- **現状**：`store()`（`WorkRecordController.php:227-289`）・`storeForMobile()`（同 156-222）・`update()`（同 352-424）の3メソッドが、**単価取得 → work_minutes算出 → WorkRecord 作成 → defects 整合性チェック → 明細作成**という同じフローを3度書いている。
- **提案**：`App\Services\WorkRecordService` を新設し、以下に集約。

  ```
  WorkRecordService
   ├─ create(array $data): WorkRecord
   ├─ update(WorkRecord $record, array $data): WorkRecord
   ├─ delete(WorkRecord $record): void
   └─ syncDefects(WorkRecord $record, array $defects): void
  ```
  - Controller 3本は Service 呼び出しに統一。
  - defects合計一致チェックは `Request` 側の `withValidator()` に寄せ、Service からは throw させない（= 422が返る正しい形に）。

### B-2. PDF生成ロジックのBuilder化

- **現状**：`ClientInvoiceController@pdf`（`ClientInvoiceController.php:254-459`、約200行）と `StaffInvoiceController@pdf`（`StaffInvoiceController.php:231-326`、約90行）がController直下にあり、**mPDFの初期化・ビューへのデータ整形・ページ分割**まで全部ここに書かれている。
- **提案**：
  - `App\Pdf\StaffInvoicePdfBuilder` と `App\Pdf\ClientInvoicePdfBuilder` を作成（`build(Invoice $invoice): string`）。
  - mPDF 初期化は `App\Pdf\MpdfFactory::makeA4Portrait()` に共通化。
  - Controller は `return response()->make($builder->build($invoice), 200, ...);` だけに減らす。
  - これにより、将来の「PDFをメール送信」「PDFを Storage に保存して過去版を参照」への展開が容易。

### B-3. `index()` の重複クエリをScope化

- **現状**：以下のクエリが5か所以上で繰り返し登場。
  - `Staff::where('active_flag', true)->with('staffType')->orderBy('name')->get()`
  - `Drawing::where('active_flag', true)->with('client')->orderBy('drawing_number')->get()`
- **提案**：`Staff` に `scopeActive()` と `scopeOrderedByName()`、`Drawing` に `scopeActive()`・`scopeOrderedByNumber()` を追加。以降は `Staff::active()->with('staffType')->orderedByName()->get()` に統一。

### B-4. カラム命名の揺れ解消：`active_flag` vs `active_flg`

- **現状**：`staff.active_flag`・`drawings.active_flag` に対し、`work_rates.active_flg` だけ綴りが違う。
- **提案**：次のマイグレーションで `work_rates.active_flg → active_flag` にリネーム（`Schema::table` の `renameColumn`）。あわせてコード内 `active_flg` を全面置換。古い単価レコードへの影響なし。

### B-5. マジックナンバー・固定文字列の外出し

| 現状の定数 | 場所 | 提案 |
|-----------|------|------|
| 消費税率 `0.1` | `StaffInvoiceService:101` / `ClientInvoiceService:62` / Controller | `config/invoice.php` に `tax_rate` として定義 |
| 残業境界 17時 | `WorkRecord.php:114` | `config/invoice.php` に `overtime_hour` として定義、将来は曜日別・祝日別にも拡張可能 |
| ページ件数 20 | 各 Controller `paginate(20)` | `config/pagination.php` |
| 客先名固定 `'株式会社○○'` | `ClientInvoiceController.php:98, 357` | **C-1で廃止**（下記） |
| 作業方法名 `'仕上げ加工'` | `ClientInvoiceController.php:315` | 実際の `workMethod->name` を使用 |

### B-6. コントローラに残る「素のtry/catch」

- **現状**：`WorkRecordController` などで、Service導入前に書かれた `DB::beginTransaction() → try → catch → back()->withErrors()` がController直下に残る。
- **提案**：Service層に移した後は、アプリ共通の例外ハンドラ（`App\Exceptions\Handler`）で `App\Exceptions\BusinessLogicException` 等を 422＋flashメッセージで返す形に統一。Controller のコードが一気に読みやすくなる。

### B-7. N+1 リスク

- **現状**：`WorkRecordController@edit`（`WorkRecordController.php:298-301`）で `$workRecord->staffInvoiceDetails` に対して `map(fn($d) => $d->staffInvoice)` を呼ぶが、**`staffInvoice` を eager load していない**。明細ごとにクエリが発行される。
- **提案**：`->with('staffInvoiceDetails.staffInvoice.staff')` を事前に宣言。

### B-8. `WorkRecordDefect` の同期ロジック

- **現状**：`WorkRecordController@update` (396行) が `$workRecord->defects()->delete()` → 再作成。不良明細数が多いとID系の副作用がある（履歴系テーブルで差分取りたい場合に不利）。
- **提案**：将来の監査ログ導入を見越して、`upsert` 方式での同期に切り替え。ただし現時点では影響軽微のため、監査ログ導入時にまとめて実施で可。

## 4. 改善提案【C：業務機能・UX】

### C-1. 客先請求書の客先選択UI化（最優先機能）

- **現状**：`ClientInvoiceController.php:97-100` が `Client::firstOrCreate(['name' => '株式会社○○'])` で固定。PDFにも `clientName = '株式会社○○'` が埋め込まれる（L357）。**複数客先へ請求書を切ることが物理的に不可能**。
- **提案**：
  - `Pages/ClientInvoices/Create.vue` に客先セレクトボックスを追加。
  - `CreateClientInvoiceRequest` に `client_id` を追加。
  - PDF側は `$clientInvoice->client->name` を使用。
  - `clients` テーブルに `postal_code` / `address` / `phone` / `invoice_registration_number`（インボイス対応）カラムを追加。

### C-2. ダッシュボードのKPI化

- **現状**：`Pages/Dashboard.vue` は各画面へのリンクカードのみ（153行のうち実質すべてナビゲーション）。
- **提案**：以下のKPIカードを追加。
  - 今月のスタッフ請求予定額 合計（draft＋fixed）
  - 今月の客先請求予定額 合計
  - 未請求の作業実績件数・重量合計
  - 当月の不良率（不良数 / 総生産数）
  - スタッフ別・今月の請求額 TOP5（棒グラフ）
  - 未処理タスク：「確定待ちのスタッフ請求書 N件」「未紐付のスタッフ請求書 N件」
- **実装**：Controller を `DashboardController@index` に切り出し、`DashboardQueryService` で集計。フロントは Chart.js か ApexCharts。

### C-3. 作業実績のCSV/Excel インポート／エクスポート

- **現状**：スマホ入力はあるが、まとめて帳票入力したい場面（ベテラン社員の紙メモをあとでまとめて入力、など）の導線がない。
- **提案**：
  - `laravel-excel` (maatwebsite/excel) を導入。
  - `/work-records/import`（ドラフト作成→確認→一括投入）と `/work-records/export`（期間・スタッフで絞って Excel 出力）。
  - 一覧画面右上に「CSVダウンロード」ボタン追加。

### C-4. 請求書のメール送付

- **現状**：PDFはダウンロードのみ。スタッフに紙を渡すか、個別メール添付運用。
- **提案**：
  - スタッフ請求書：`staff.email` 宛に PDF添付＋所定文面で送信する `Mail\StaffInvoiceMail`。
  - 客先請求書：客先マスタに担当者メールアドレスを複数保持できるよう `client_contacts` テーブルを追加（名前・メール・CC区分）。
  - Laravel Mail 経由、送信履歴は `mail_logs` に記録。
- 参考：既存の `queue` は `sync` なので、まずは同期送信でOK。負荷がかかるようなら Redis Queue に切り替え。

### C-5. インボイス（適格請求書）対応

- **現状**：`staff.tax_type`（taxable/tax_exempt）はあるが、**登録番号カラムなし**。2026年現在、インボイス制度上の要件を満たしていないリスクが高い。
- **提案**：
  - `staff` に `invoice_registration_number`（nullable, `T`+13桁）、`clients` にも同カラムを追加。
  - PDFテンプレート（`resources/views/pdf/staff-invoice.blade.php` / `client-invoice.blade.php`）に「発行者登録番号」欄と「税率ごとの税額」欄を追加。
  - スタッフマスタ編集画面で登録番号を入力。

### C-6. マスタ画面に「Show」を追加

- **現状**：`Pages/Clients`・`Drawings`・`Staff`・`WorkRates`・`WorkMethods`・`DefectTypes` に `Show.vue` が無く、一覧から直接 `Edit` に飛ぶ。閲覧専用の一般ユーザー（将来の一般スタッフ）にとって不便。
- **提案**：各マスタに `Show.vue` と `show` ルートを追加。編集ボタンは `admin_flg=true` のみ表示。

### C-7. 検索・フィルタUIの統一

- **現状**：一覧画面ごとにフィルタ項目がバラバラ（作業実績＝staff/drawing/method/期間、単価＝search/drawing/method、客先請求書＝client/status/期間）。
- **提案**：共通の `<FilterBar>` Vueコンポーネントを作成。フィールドはPropsで定義できる形に。クリアボタン・絞り込み中バッジも統一。

### C-8. スマホ入力の改善

- **現状**：`Pages/Mobile/WorkRecords/Create.vue` は図番の部分一致検索で1件なら自動選択（L63-69）まで作り込んでいるが、**前回入力の記憶**や**作業中タイマー**がない。
- **提案**（優先度順）：
  1. 図番・作業方法の「直近使った順」ソート（localStorage に最後の5件を記憶）。
  2. 開始ボタン／終了ボタン形式（現在時刻を自動セット、5分単位丸め）。誤タップ防止のため **10秒のアンドゥ** を付ける。
  3. PWA化（ホーム画面追加、オフライン時は IndexedDB 保存→復帰時に同期）。
  4. カメラ連携（不良品写真を `work_records.photo_path` に添付）。

### C-9. 一覧の情報量不足

- **現状**：`Pages/WorkRecords/Index.vue` 他、一覧の列が最低限で「合計重量」「金額見込み」が出ていない。
- **提案**：各一覧に「合計重量」「想定請求額」を列追加。ソート可能に。

## 5. 改善提案【D：テスト・CI／EoL対応】

### D-1. Laravel 9 のEoL対応（最優先）

- **現状**：`composer.json` で `"laravel/framework": "^9.19"`。Laravel 9 は **2024年2月にセキュリティサポート終了済**。現状稼働中の本番はセキュリティパッチが降ってこない。
- **提案**：
  - **Step 1**：Laravel 10 へ（PHP 8.1以上必須）。`shift.laravel.tools` や Upgrade Guide を使って 2〜3日工数。
  - **Step 2**：Laravel 11 へ（PHP 8.2以上、構造が変わるので小さめのPR多数）。
  - **Step 3**：Inertia 0.6.8 → 2.x（SSRやフォーム更新APIが改善）。
  - **Step 4**：`laravel/breeze` 1.19 → 最新。
- 事前にテスト（D-3）を拡充しておくと安全。

### D-2. PHP 8.0 → 8.2/8.3 への更新

- **現状**：`composer.json` の `"php": "^8.0.2"`。PHP 8.0 は **2023年11月にEoL**。
- **提案**：VPSのPHPを 8.2 系に上げ、`composer.json` を `"php": "^8.2"` に。Dockerfile がある場合はベースイメージも更新。

### D-3. テスト追加（Service単位から）

- **現状**：`tests/Feature` は `ExampleTest` と `ProfileTest`、`Unit` は `ExampleTest` のみ。ビジネスロジックのテストがゼロ。
- **提案**：優先度の高いものから追加。
  1. `StaffInvoiceServiceTest`：createInvoice の subtotal/tax/total 計算、非課税スタッフの tax=null、空期間で例外、スナップショット不変性（単価変更後に再読んでも金額が変わらないか）。
  2. `ClientInvoiceServiceTest`：updateAdjustment の total 再計算、fix済の解除不可、複数 StaffInvoice の合計。
  3. `WorkRateStoreTest`：新単価追加時に既存の `effective_to` が前日に閉じられること。
  4. `MobileAuthTest`：A-2対応後のトークン認証のテスト。

### D-4. CI（GitHub Actions）の整備

- **現状**：リポジトリに `.github/workflows/` が存在するか確認要だが、READMEからは読み取れず。
- **提案**：`ci.yml` で以下を main／develop へのPush時に実行。
  - `composer install` → `php artisan test`
  - `npm ci` → `npm run build`
  - `./vendor/bin/pint --test`（整形差分チェック）
  - `php artisan migrate --pretend` で破壊的マイグレーション検知。

### D-5. 不要依存の削除

- **現状**：`composer.json` に `barryvdh/laravel-dompdf` があるが、実際はmPDFのみ使用（手順書 2章にも明記）。
- **提案**：dompdf を `composer remove`。`composer.lock` が軽くなる。

## 6. 改善提案【E：運用・監査】

### E-1. 操作ログ（Audit Log）

- **現状**：誰がいつ請求書を確定した／単価を変更した、の履歴が残らない。
- **提案**：`owen-it/laravel-auditing` を導入。`StaffInvoice`・`ClientInvoice`・`WorkRate`・`Drawing`・`Staff` に Auditable trait を追加。`audits` テーブルで before/after を自動記録。

### E-2. ソフトデリート

- **現状**：`deleteWorkRecord` などで物理削除。誤削除時のリカバリができない。
- **提案**：`WorkRecord`・`StaffInvoice`・`ClientInvoice`・マスタ系に `SoftDeletes`（`deleted_at`）を追加。復旧用に `/admin/trash` 画面（admin_flg限定）。

### E-3. バックアップ自動化

- **現状**：DB バックアップ運用の記述が `手順書.md` にない。
- **提案**：`spatie/laravel-backup` を導入。日次でDB＋`storage/app/public`（図番画像）を圧縮し、S3 or ローカル別ディスクへ。`システム開発部門/VPS管理/160.251.139.12/` 側に手順書を残す。

### E-4. エラーモニタリング

- **現状**：`spatie/laravel-ignition` は入っているが本番エラーの集約先がない。
- **提案**：Sentry もしくは Bugsnag を導入し、本番で発生した例外をSlack通知。スタッフが「エラーが出たが誰にも言わなかった」問題を防げる。

### E-5. タイムゾーン運用の明文化

- **現状**：`work_records.start_time/end_time` は datetime（TZなし）、`payment_due_date` のみ UTC 0:00 という混在（手順書 9.6 参照）。
- **提案**：`config/app.php` の `'timezone' => 'Asia/Tokyo'` が設定されているかを確認。新規の日時カラムを追加する際のルール（datetime で JST、日付のみは date 型）を `手順書.md` に明文化。

## 7. 実行計画（Phase 分割）

優先度・工数・依存関係を考慮した推奨順。工数は1人日ベース。

### Phase 1：止血とガード（2週間・3〜5人日）
| No | 施策 | 参照 | 工数 |
