## 1. システム概要

### 1.1 目的

平田工業（鋳物仕上げ加工業）の現場作業から請求業務までを一気通貫で電子化する社内Webアプリケーション。スタッフが図番・作業方法ごとに作業実績（時間・出来高・不良）を記録し、その実績を期間締めで集計してスタッフ向け請求書（社外発注スタッフへの支払）と客先向け請求書（委託元への請求）を自動生成・PDF発行する。

### 1.2 業務領域

| 領域 | 内容 |
|------|------|
| 作業実績管理 | スタッフが図番・作業方法ごとに作業時間と出来高（良品/不良）を記録。不良は内訳（不良種別×数量）でブレークダウンして記録。 |
| スタッフ請求管理（スタッフ→自社） | 期間締めで作業実績を集計し、スタッフ個別の請求書を自動生成・PDF発行。正社員/個人事業主で単価体系が異なる。 |
| 客先請求管理（自社→委託元） | 確定済みのスタッフ請求書を束ねて客先請求書を生成。差額調整・消費税込みで一括PDF出力。 |
| マスタ管理 | 客先・図番・作業方法・作業単価（履歴管理）・スタッフ・不良種類の6種マスタを管理。 |
| スマホ入力 | 作業者が現場からスマホで作業実績を入力する専用レイアウト。QR/トークンで自動ログイン。 |

### 1.3 利用者と権限

| 利用者 | 権限 | 主な操作 |
|--------|------|----------|
| 管理者スタッフ（`admin_flg=true`、現状は平田 敦士のみ） | 全機能 | マスタ管理・請求書発行・全実績の管理 |
| 一般スタッフ | 限定（※認可は現状未実装。改善提案 A-3 参照） | 作業実績の登録・閲覧 |
| 現場作業者（スマホ） | 作業実績入力のみ | スマホ画面から自分の作業実績登録 |

> 注：FormRequest の `authorize()` はすべて `return true;` であり、ロール別の認可（Policy）は未実装。`admin_flg` カラムは存在するが権限制御には未使用。Phase 2 で Policy 整備予定（改善提案書 A-3）。

## 5. 業務ロジック仕様

### 5.1 作業単価の履歴管理

- `work_rates` は「図番 × 作業方法 × 期間」で単価を保持。`effective_from` / `effective_to` で有効期間を管理し、**過去分は改変しない**。
- 新単価を登録すると `WorkRateController@store` が既存の有効単価の `effective_to` を新単価の前日に更新（期間の自動閉鎖）。
- 有効単価の取得は `Drawing::getEffectiveWorkRate()`：
  - 条件：`work_method_id` 一致 AND `active_flg=true` AND `effective_from ≤ 指定日 ≤ effective_to（or NULL）`
  - 複数該当時は `effective_from` 降順で最新を採用。
- `active_flg=false` の単価に紐づいた作業実績は `RelinkWorkRecordsService` で有効単価に貼り直し可能（`POST /work-rates/relink-work-records`、dry-run対応）。

### 5.2 作業実績の登録ルール

- 登録時、`start_time` 時点で有効な `work_rate` を自動取得し `work_rate_id` に保存（**単価スナップショット**）。
- `work_minutes` = `end_time - start_time` の分数で自動算出。
- スマホ入力では分を5分単位に丸め、0〜5時は入力不可（始業前の誤登録防止）。
- 不良数（`quantity_ng`）の合計と不良内訳（`defects[*].defect_quantity`）の合計が**一致しないと例外**。
- 残業判定 `isOvertime()`：開始時刻の hour ≥ 17 なら残業（個人事業主の残業単価に使用）。
- 請求書明細（`StaffInvoiceDetail`）が既に存在する作業実績（`isInvoiced()=true`）は**編集・削除不可**。
- 更新時は既存 `defects()` を全削除→再作成で同期。

### 5.3 スタッフ請求書（StaffInvoiceService）

**対象実績の抽出**（`createInvoice(staffId, periodFrom, periodTo)`）：
- `staff_id` 一致 かつ `start_time` が期間内
- `whereDoesntHave('staffInvoiceDetails')`（**未請求**のみ）
- `whereHas('workRate', active_flg=true)`（有効単価に紐づくもののみ）

**金額計算（新方式：2026-05 以降に統一）**：
```
個単価   unitPiece = round(weight_per_unit × kg単価)
明細金額 amount    = total_quantity × unitPiece
```
- 「kg単価」は `WorkRate::getRateForStaff($staff, $isOvertime)` で決定：
  - 正社員 → `rate_employee`
  - 個人事業主（通常）→ `rate_contractor`
  - 個人事業主（残業）→ `rate_overtime`
- 明細には `quantity`（重量kg）・`unit_price`（kg単価）・`amount`（計算済金額）を**スナップショット保存**。

> 旧方式は `amount = total_weight × unit_price`（重量×kg単価）。新方式は個単価を先に四捨五入してから数量を掛けるため、丸め処理の差で金額が変わりうる。**既存明細のDB値は変更せず、新規請求書のみ新方式**。

**消費税・合計**：
```
subtotal = Σ(amount)
tax      = (tax_type=taxable) ? floor(subtotal × 0.1) : null
total    = subtotal + (tax ?? 0)
```
- 税率10%固定、消費税は `floor`（切り捨て）。

**請求書番号**：`STAFF-YYYY-NNN`（年単位の連番、3桁ゼロ埋め）。

**状態遷移**：`draft → fixed → paid`
- `fix`：draft→fixed。`unfix`：客先請求書に紐付いていない場合のみ fixed→draft。
- 削除：客先請求書に紐付いていなければ可。

### 5.4 客先請求書（ClientInvoiceService）

**作成**（`createInvoice(clientId, staffInvoiceIds[], periodFrom, periodTo)`）：
- 対象は `status=fixed` のスタッフ請求書のみ。
- 既に他の客先請求書に紐付くスタッフ請求書が含まれていれば例外（二重請求防止）。
- **客先は画面のドロップダウンで選択**（最新仕様。旧「株式会社○○ 固定」は廃止済）。

**DB保存時の金額**：
```
subtotal = Σ(staffInvoices.total)
tax      = floor(subtotal × 0.1)
total    = subtotal + adjustment_amount + tax
```

**PDF/画面表示時の金額**（DBとは別に再計算）：
```
行金額 amount = quantity × round(weight_per_unit × rate_employee)   ← 常に rate_employee 固定
subtotalForInvoice = Σ(行金額)
taxForInvoice      = floor(subtotalForInvoice × 0.1)
totalForInvoice    = subtotalForInvoice + taxForInvoice + adjustment_amount
```
> 表示は `rate_employee` 固定で再計算するため、DBの `subtotal` と表示上の小計が一致しない場合がある。DB値は互換性のため変更しない。

**差額調整**（`updateAdjustment`）：draft時のみ。`total = subtotal + adjustment_amount + tax` を再計算。

**状態遷移**：`draft → fixed → issued`
- `fix`/`unfix` で draft↔fixed。
- PDF出力は `fixed`/`issued` のみ可。削除は `draft` のみ可。

**請求書番号**：`CLIENT-YYYY-NNN`。

### 5.5 支払期限（payment_due_date）

- 保存は UTC 0:00、表示は `format('Y-m-d')`。これにより JSON化時の前日ズレを防止。
- draft時のみ変更可。**日付系カラムを追加する際はこの方式に倣うこと。**

### 5.6 重要な定数・丸めルール

| 項目 | 値 | 箇所 |
|------|-----|------|
| 消費税率 | 10%（`0.1` 固定） | StaffInvoiceService / ClientInvoiceService |
| 消費税の丸め | `floor`（切り捨て） | 同上 |
| 個単価の丸め | `round`（四捨五入） | 金額計算（新方式） |
| 残業開始時刻 | 17:00（`hour >= 17`） | WorkRecord::isOvertime() |
| スマホトークン長 | 48文字ランダム | Staff::regenerateMobileLoginToken() |
| ページ件数 | 20件/ページ | 各 Controller `paginate(20)` |

> 税率・残業境界・ページ件数はコード内ハードコード。`config/invoice.php` 等への外出しが改善提案 B-5 で提案されている。

## 8. 認証・セキュリティ仕様

- **ガード**：`config/auth.php` の guard `web` / provider `staff` → **`users` ではなく `staff` テーブルで認証**。
- `Staff` は `Authenticatable` を継承、`getAuthIdentifierName()` で `login_id` を返す。パスワードは bcrypt。
- **スマホ自動ログイン**（`WorkRecordController@createForMobile`）：未ログイン時、`?token=`（`mobile_login_token` 一致・`active_flag=true`）または `?staff_id=`（同条件）で該当スタッフを `Auth::guard('web')->login()`。`?staff_id=` 方式はパスワードレスのため、トークン方式への移行と IP 制限が推奨（改善提案 A-2）。
- 既知のセキュリティ留意点（詳細は改善提案書）：
  - `?staff_id=` パスワードレスログイン（A-2）
  - FormRequest `authorize()` が全て `true` で認可未実装（A-3）
  - `/register` 等 Breeze ルートが開放（A-4）
  - `/__debug` ルート（本番では未登録ガード済だが残存）（A-1）

## 9. PDF 出力仕様

- 生成エンジンは mPDF 8.2（日本語フォント自動適用）、A4 縦。
- **スタッフ請求書**（`pdf/staff-invoice.blade.php`）：1スタッフ1ファイル。明細は `日付｜客先｜品番｜品名｜実績数｜個単価｜金額`。金額はスナップショット（`detail.amount`）を優先使用。
- **客先請求書**（`pdf/client-invoice.blade.php`）：1本のテンプレで `type` 切替し複数ページを描画。
  - 1枚目：スタッフごとに集約した金額リスト（`仕上げ加工 + スタッフ名` 行）。
  - 2枚目以降：スタッフごとの明細内訳。
  - 金額は表示時に `quantity × round(weight_per_unit × rate_employee)` で再計算（5.4参照）。

## 13. 追加開発ガイドライン

### 13.1 不変条件（壊してはいけないもの）
- [ ] **請求書のスナップショット**：`staff_invoice_details.unit_price/amount`、`work_records.work_rate_id` は作成時点の固定値。過去請求書の金額を再計算しない。
- [ ] **isInvoiced ガード**：請求書明細を持つ作業実績は編集・削除不可。
- [ ] **ステータス遷移**：スタッフ請求書 `draft→fixed→paid`、客先請求書 `draft→fixed→issued`。確定後の編集・確定済からの不正な削除を許さない。
- [ ] **単価履歴**：`effective_from/to` の過去レコードは改変しない。新単価追加時は既存の `effective_to` を前日で閉じる。

### 13.2 実装規約
- [ ] 日時カラムを追加する場合：作業時刻系は `datetime`（JST、TZなし）、日付のみは `date`、支払期限的なものは UTC 0:00 保存＋表示時 `format('Y-m-d')` 方式に合わせる。

- [ ] ルート名・Ziggy 依存の Vue 側参照を grep で確認。
